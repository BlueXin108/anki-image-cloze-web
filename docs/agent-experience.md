# Agent 经验与环境约束

## 这份文档的目的

这份文档记录“不是业务代码本身的问题，但会影响开发效率和判断”的环境经验，帮助后续 agent 在同样环境下少走弯路。

## 2026-04-01：当前 Windows 环境会偶发误拦正常工具操作

### 典型现象

- 读文件时偶发失败。
- `apply_patch` 偶发失败。
- 明明是普通读取或普通小补丁，却报出和 helper、防火墙、沙箱相关的异常。

常见表现类似：

- `helper_firewall_rule_create_or_add_failed`
- `SetRemoteAddresses failed`
- 一部分命令能执行，另一部分看起来随机失败

### 当前判断

这类问题大概率不是业务代码错误，而是当前 Windows 沙箱、helper 或相关防火墙规则偶发误伤正常操作。

换句话说：

- 失败现象是真实存在的
- 但“失败”不一定说明代码逻辑有问题
- 先把它当作环境层噪声，再决定是否继续怀疑代码

### 目前最稳的应对方式

1. **大补丁拆小补丁**
   - 不要一次改很多文件或一次改很大一段内容。
   - 优先按文件、按功能点、按区块逐步落盘。

2. **先读上下文，再小步修改**
   - 先确认真实内容，再改最小必要范围。
   - 不要在没有读准上下文时做大范围替换。

3. **每做一小步就验证**
   - 读回文件内容
   - 跑最小检查
   - 再继续下一步

4. **普通读取被拦时，先换更小粒度**
   - 先缩小读取范围
   - 再考虑是否需要提权读取
   - 不要一上来就把所有操作都变成重型方案

5. **不要把环境噪声误判成业务 bug**
   - 如果失败像是随机的、与代码改动不强相关、且错误信息偏 helper/防火墙/沙箱层，先考虑环境因素

## 推荐工作方式

### 写文档

- 优先小步更新
- 一份文档一份文档处理
- 改完就读回确认

### 改代码

- 优先原子化补丁
- 共享逻辑先改最底层，再改界面层
- 涉及前后端联动时，分层验证

### 改 UI

- 先回顾当前项目里已经有的可复用组件
- 优先考虑：
  - 直接复用已有组件
  - 给已有组件增加可配置项
  - 用小范围样式渗透完成细调
- 不要为了一次局部界面改动，重复搓出一套功能近似的新组件
- 做弹窗类界面时，先检查底层组件有没有默认宽度限制
- 需要更宽的桌面端弹窗时，优先参考当前项目里已经正常工作的其他弹窗写法，再决定覆写哪些 `max-w-*`

### 遇到异常时

按下面顺序判断：

1. 是不是当前环境误拦？
2. 是不是补丁太大？
3. 是不是读写范围太大？
4. 确认不是以上问题后，再重点怀疑业务代码

## 当前结论

到目前为止，还没有在仓库内部找到一个“永久关闭这类误拦”的根治办法。

**当前最稳定的方案不是彻底消除它，而是采用更稳的工作策略：**

- 小补丁
- 高频验证
- 环境异常与业务 bug 分开判断

这条经验应长期保留，直到开发环境被整体替换或确认已经稳定。
