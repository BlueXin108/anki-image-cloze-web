# Git 与工作树说明

## 这份文档解决什么问题

这份文档专门回答三个问题：

1. 这个项目当前的 Git 状态到底是什么
2. 后续应该怎样做提交、分支和工作树
3. 如果要在保留当前本地版的前提下，继续做一个 Web / Server 独立版本，最稳的路线是什么

## 当前真实状态

截至目前，这个项目目录本身**还不是独立 Git 仓库**。

当前 Git 根目录在上一级：

- `E:/Dev_Env/Projects/AIBuild/Tools`

这意味着：

- 当前项目只是上一级仓库里的一个子目录
- 这个项目自己还没有独立提交历史
- 因此直接从当前项目目录做 `worktree`，会遇到：
  - `fatal: invalid reference: HEAD`

## 为什么会报 `invalid reference: HEAD`

出现这个错误，通常不是 `git worktree` 功能坏了，而是当前前提不成立。

在这个项目里，原因有两层：

1. 当前目录不是独立仓库根目录
2. 当前这条线没有真实提交，所以没有可用的 `HEAD`

一句话理解：

**没有基线提交，就没有可用的 `HEAD`；没有独立仓库边界，后面维护会越来越乱。**

## 这个项目的推荐路线

对于当前项目，推荐采用下面这条路线：

1. 先把这个项目整理成独立 Git 仓库
2. 做一次基线提交，固定当前“本地版”状态
3. 把当前本地版作为稳定线继续维护
4. 再开一条新分支，专门做 Web / Server 独立版本
5. 如果需要长期并行开发，再为新分支创建永久工作树

## 为什么先独立仓库，再谈 worktree

因为 worktree 适合“一个已经成形的仓库，开第二个并行工作目录”。

而你现在的场景更像是：

- 手上已经有一个项目
- 但它还没建立自己的 Git 历史
- 还准备同时维护本地版和 Web 版

如果这时直接用上一级大仓库硬开 worktree，后面会有这些问题：

- 这个项目和上级其他内容会缠在一起
- 分支、提交、工作树边界会不清楚
- 将来查历史、对比变化、单独迁移都会麻烦

所以先独立仓库，是最重要的一步。

## 提交规范

### 最重要的原则

**一次提交只做一类事情。**

不要把不相关的改动混在一个提交里。

例如，下面这样是好的：

- 一次只修编辑器交互
- 一次只改导出模板
- 一次只改文档

下面这样就不好：

- 同时改遮罩逻辑、OCR、导出 UI、文档，再一次性提交

### 推荐提交前缀

建议统一使用这些前缀：

- `feat`
  - 新功能
- `fix`
  - 修 bug
- `refactor`
  - 重构，但不改变主要行为目标
- `docs`
  - 文档变更
- `chore`
  - 工程整理、脚本、配置等杂务

### 推荐提交信息格式

可以用：

- `feat: 新增手动模式卡片分组`
- `fix: 修复聚焦模式遮罩漂移`
- `docs: 中文化项目文档并补充 agent 经验`
- `chore: 初始化独立仓库基线`

关键不是格式花哨，而是：

- 一眼看懂做了什么
- 以后回看能快速理解

## 分支规范

### 稳定主线

当前本地版建议保留一条稳定主线，例如：

- `main`

### 大改新线

如果要做 Web / Server 独立版本，建议新开明确名称，例如：

- `codex/web-server-rebuild`

为什么名字要明确：

- 一眼能看出它不是普通小修
- 也能看出它和当前本地版是平行演进关系

## 工作树适用场景

### 什么时候该用 worktree

适合以下情况：

- 两条版本线要长期并行
- 不想频繁切分支
- 本地版还要维护，同时新版本要大改

### 什么时候不用 worktree

如果只是：

- 小功能试验
- 短期修改
- 很快就会回到主线

那普通分支通常就够了。

## 结合本项目的最佳实践

你的目标是：

- 保留当前本地版
- 再做一个 Web / Server 独立版
- 两者隔离

这正是 worktree 最适合的场景。

推荐最终形态是：

- 当前目录
  - 本地版主工作区
- 新工作树目录
  - Web / Server 独立版工作区

## 后续实际执行建议

建议按这个顺序推进：

1. 把当前项目整理成独立仓库
2. 完成第一次基线提交
3. 确认当前版本能正常启动
4. 新开 Web / Server 分支
5. 需要长期并行时，再创建永久工作树

## 维护者提醒

后续 agent 在处理 Git 相关任务时，应先检查：

1. 当前项目是否已经是独立仓库
2. 当前仓库是否已有真实提交
3. 当前操作是小修，还是值得单独开分支 / 工作树的大改

如果这三件事没先看清，就很容易误判问题本质。
