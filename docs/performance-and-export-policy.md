# 性能与导出策略

这份文档用于约束后续维护时的统一标准，重点回答两件事：

1. 什么地方可以为了速度做缓存和降载
2. 什么地方不能为了省事而牺牲最终导出语义

## 总原则

- 编辑区和下方预览属于前端本地预览链路，目标是稳、快、跟手。
- 最终导出属于成品链路，目标是结果语义准确，而不是只求省一次重算。
- 任何时候都不要把“已经压缩过的中间结果”伪装成“重新生成的高质量导出”。

## 预览策略

### 当前统一规则

- 原图解码允许缓存。
- 问题面 / 答案面预览结果允许缓存。
- 预览缓存的键必须至少包含：
  - 当前图片来源
  - 当前裁切范围
  - 当前遮罩列表
  - 当前选中的卡片分组
  - 当前导出格式
  - 当前导出质量
- 预览缓存必须使用有上限的淘汰策略，避免无限增长。
- 当前实现采用：
  - 最多保留 48 组预览结果
  - 总体积上限约 64MB

### 为什么这样做

- 编辑时最容易卡的，不一定是保存，而是下方预览反复重画。
- 同一张图、同一组遮罩、同一输出格式，如果结果没变，就不应该继续重算。
- 预览结果现在统一走 `Blob + object URL`，不再默认走大段 base64 字符串，减少内存和字符串分配压力。

### 维护约束

- 不要再把下方预览改回“每次状态一动就重新生成 dataURL”的模式。
- 不要引入无上限缓存。
- 如果后续再加预览维度，例如描边风格、背景模式、文字标注，必须一起进入缓存键。
- 预览可以降优先级、延后更新，但不能和最终导出共用“预览级中间产物”。

## PWA 缓存策略

### 当前统一规则

- 这是工具型网页，PWA 缓存策略应当偏激进，优先减少重复加载。
- 构建产物里的前端资源要在安装阶段整体预缓存：
  - 主入口 HTML
  - 构建后的 JS
  - 构建后的 CSS
  - 懒加载分包
  - 字体、图标和其他静态资源
- 页面再次打开时，优先直接使用本地缓存的前端资源。
- 有新版本时，再通过更新提示引导用户刷新切换。

### 为什么这样做

- 这个项目不是内容站，而是工具应用。
- 用户会频繁重复打开，而且会进入多个编辑、导出、设置模块。
- 如果每次打开都重新拉一遍整套前端资源，体感会明显偏笨重。
- 导出确认页属于高频路径，当前已并回主包，避免再把“打开导出页”本身变成一次额外等待。
- 真正打包 APKG / 图像组所需的导出器模块，会在用户开始编辑后做一次后台预热，减少点下“开始导出”时的等待。

### 不能缓存的内容

- 用户导入的图片项目
- 浏览器本地项目数据
- `AnkiConnect` 相关实时请求
- 任何需要实时反映本机状态的连接结果

### 维护约束

- PWA 缓存应当主要服务于“前端代码和静态资源”，不要越界接管项目数据。
- 新增懒加载模块或静态资源时，要确认它会被纳入预缓存清单。
- 更新策略优先保证“旧版本可稳定继续用”，不要为了抢新版本把当前会话直接打断。

## 导出策略

### 当前统一规则

- 导出前先判断图片来源状态，而不是只看设置开关。
- 每张图都要记录自己的来源状态：
  - `original`
    - 当前仍是原图链路，可开放 `WebP / JPG / PNG`
  - `import-compressed`
    - 导入时已经压缩过，只保留 `WebP`
  - `project-optimized`
    - 中途执行过项目压缩，只保留 `WebP`
  - `legacy-unknown`
    - 来自旧版本地项目、无法严格确认是否仍是原图，只保留 `WebP`

### 为什么这样做

- 如果图片早就被压缩成 WebP 或 JPG，再让用户选 PNG，只是在把压缩过的图换个壳。
- 这种“表面是 PNG，实际不是原图链路”的结果，会误导用户。
- 所以 PNG / JPG 只有在明确仍持有原图链路时才开放。

### APKG / AnkiConnect / 图像组 的统一要求

- 三条导出路径都遵守同一套格式开放规则。
- 只要当前这批图片里出现以下任一情况，就统一锁住 `PNG / JPG`：
  - 导入时启用了压缩
  - 中途执行过项目压缩
  - 来自旧版记录，无法确认是否仍是原图
- 被锁住时，界面必须明确告诉用户原因，不能只灰掉不解释。

## PNG 导出规则

### 严格要求

- 当用户选择 `PNG` 导出时，必须从当前项目持有的原图链路重新生成。
- 不能拿已经生成过的 `WebP / JPG` 预览结果再转一遍 `PNG`。
- 不能拿已经压缩过的项目图片再伪装成“PNG 成品”。

### 维护约束

- 预览缓存只服务于预览和可复用的同源输出，不代表可以跳过导出语义检查。
- 如果后续新增导出器，先确认它吃的是原图链路还是预览级结果。
- 只要导出格式是 `PNG`，就必须重点回看这条约束有没有被绕开。

## 本地保存与来源标记

- 浏览器本地保存必须继续区分：
  - 图片资源
  - 编辑元数据
- 每次改单张图时，不要重写整批图片资源。
- 图片来源状态要跟随项目一起保存和恢复。
- 旧版恢复记录如果拿不到这层来源信息，必须保守处理，不要默认当作原图。

## 移动端交互兜底

### 当前统一规则

- 移动端常规页默认只读，真正编辑集中在聚焦编辑层。
- 移动端每次重新打开聚焦编辑时，都会重建内部编辑实例。
- 这份重建只清掉临时交互状态，不清掉已经提交过的遮罩和裁切结果。

### 为什么这样做

- 移动端浏览器的触摸、长按、页面缩放和组件内部拖动容易互相影响。
- 真正根治这类问题往往比桌面端更难，也更依赖具体浏览器。
- 所以当前先保留一个稳定兜底：关闭再打开聚焦层时，至少能拿到一份干净的编辑实例，不必先刷新整页。

### 维护约束

- 这个兜底只对移动端生效，不要顺手把桌面端也改成每次重建。
- 若后续继续改移动端手势，优先保证这个“关闭后局部复位”的安全出口不要丢。
- 如果后续真的彻底解决了移动端那类残留状态问题，再决定是否保留这层重建策略。

## 维护清单

以后凡是改到以下任一模块，都要回看这份文档：

- `manual-preview`
- `image-group-export`
- `apkg-export`
- `anki-connect`
- `project-store`
- 导入预处理
- 项目压缩
- 导出页格式选择

每次改动后，至少检查：

1. 下方预览是否仍然命中缓存，而不是频繁整块重画
2. PNG / JPG 是否只在原图链路下开放
3. APKG / AnkiConnect / 图像组三条导出路径是否保持同一套开放规则
4. 中途执行项目压缩后，导出页是否立即锁住不再合规的格式
5. 旧项目恢复后，是否仍然走保守策略，避免误把未知来源当原图
