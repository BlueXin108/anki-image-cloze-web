# Vercel 观测接入说明

## 当前已完成

当前前端已经同时接入了这两层：

- `@vercel/analytics`
  - 负责页面访问和自定义事件
- `@vercel/speed-insights`
  - 负责真实用户性能数据

这次不再只看“有多少访问”，而是把工具型网页最关心的几类数据一起接了起来：

- 页面访问和设备分布
- 真实用户的速度表现
- 导入、编辑、导出这些关键流程的耗时
- 慢保存告警出现的频率
- 聚焦编辑、导出流程、按钮点击这些高频交互

## 代码位置

- 依赖安装：
  - `frontend/package.json`
- 入口挂载：
  - `frontend/src/main.tsx`
- 统一事件工具：
  - `frontend/src/lib/telemetry.ts`

## 现在会记录什么

### 基础访问

- 页面访问量
- 路径级访问数据
- 来源、设备、地区等基础访问画像

### 真实速度

- 首次内容出现速度
- 主要内容出现速度
- 交互响应速度
- 页面抖动情况

这一层主要看 `Speed Insights`。

### 关键业务动作

当前已经补上的重点事件包括：

- 导入图片完成
- 打开导出流程
- 导出完成
  - `AnkiConnect`
  - `APKG`
  - `图像组`
- 单张图片编辑阶段摘要
  - 停留时长
  - 遮罩总数
  - 遮罩相关操作次数
  - 裁切操作次数
  - 打开 / 关闭聚焦编辑的次数
- 工作台整段会话摘要
  - 停留时长
  - 从打开网页到导入图片的时间
  - 从导入到第一次编辑的时间
  - 从导入到打开导出流程的时间
  - 从导入到真正导出的时间
- 慢保存信号
- 慢保存提示窗真正弹出的次数

### 所有按钮 / 组件点击频次

这次没有把每一次点击都立刻单独上报，而是做成了“先在前端汇总，再分批上报”：

- 这样更适合工具应用这种高频操作场景
- 能看出哪些按钮最常被点
- 不会因为点得太频繁，把统计噪声放得很大

当前会按区域和控件类型做分类，例如：

- 页头
- 图片选择区
- 编辑区
- 聚焦编辑
- 导出流程
- 牌组浏览器

## 去哪里看

部署到 Vercel 后，建议优先从下面这个入口开始看：

- `Observability`

在这个总入口里，再分别看：

- `Web Analytics`
  - 看访问量、来源、设备、事件
- `Speed Insights`
  - 看速度、交互响应、页面稳定性

也就是说，后面不要再只盯着旧的 `Analytics` 单页了。更合适的习惯是：

1. 先进 `Observability`
2. 再按“访问 / 性能 / 事件”分开看

## 目前最值得关注的几个指标

### 速度侧

- `INP`
  - 最接近“用户觉得卡不卡”
- `LCP`
  - 最接近“打开时主内容出来得快不快”

### 产品侧

- 慢保存提示出现次数
- 导入完成到第一次编辑的时间
- 导入完成到导出的时间
- 每张图的平均编辑时长
- 每张图平均产生多少个遮罩
- 哪些按钮和导出方式最常被使用

## 当前维护规则

- 记录业务统计时，优先记“动作类别”和“计数”，不要滥记原始文件名。
- 高点击频动作优先做前端聚合，不要每点一次就直接上报一次。
- 跟图片相关的摘要，优先记录：
  - 编辑时长
  - 遮罩数量
  - 操作次数
  - 聚焦编辑次数
- 跟性能相关的摘要，优先记录：
  - 动作耗时
  - 是否触发慢保存提示
  - 压缩前后是否仍反复触发

## 使用前提

- 站点需要部署在 Vercel 上
- 需要有真实访问数据进入
- 自定义事件和速度数据主要在部署后有价值

## 后续还可以继续补什么

如果后面还想往前走，优先级建议是：

1. 把导入来源再细分
   - 系统相册
   - 文件管理器
   - 文件夹导入
2. 给导出失败增加失败类型分类
   - Anki 未连接
   - 模板问题
   - 浏览器下载失败
3. 如果后面升级到更完整的方案，再考虑把日志和观测数据接到外部平台统一留档
