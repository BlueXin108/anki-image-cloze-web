# 踩坑与纠偏记录

## 2026-04-08：浏览器里的 APKG 导出，不能直接拼装旧库的分散模块

### 现象

- APKG 导出在浏览器里报 `process is not defined`
- 继续往下补后，又会冒出 `events`、`util`、`stream` 这类浏览器不兼容报错
- 表面上像是“只差补几个兼容垫片”，实际会越补越乱

### 纠偏

- 浏览器导出优先使用项目内可控的一套最小实现，只保留真正需要的卡片、牌组、模板和打包逻辑
- SQL 的 wasm 文件必须走项目本地资源，不能依赖运行时再去外网拉取
- 不要把旧库里给 Node 或打包器准备的零散模块重新拼回浏览器链路，否则很容易再次把不兼容依赖拖进前端

## 2026-04-08：React 19 下的引用拼接，不能一边求稳一边继续传递清理链

### 现象

- 页面里某些基于 Radix 的弹层、折叠区或按钮，点击后会直接报 `Maximum update depth exceeded`
- 报错栈会落到项目自定义的 `compose-refs-fix.ts`
- 看起来像是明明已经做了 React 19 兼容补丁，却仍然在挂载阶段反复触发更新

### 纠偏

- 这层补丁除了要保证“引用回调本身稳定”，还要避免把嵌套出来的清理链继续往外传
- 在当前项目这套组件组合里，只要清理链继续参与引用切换，就可能再次触发挂载和更新来回打转
- 这里优先采用更保守的做法：只稳定分发引用，不额外转发清理函数

## 2026-04-06：PWA 只能缓存应用壳，不能顺手接管本地项目数据

### 现象

- 给网页加上 PWA 之后，很容易顺手把“离线缓存”和“项目恢复”混成一层
- 一旦把用户导入的图片、浏览器本地项目，甚至 `AnkiConnect` 请求也塞进 Service Worker
- 表面上像是更完整，实际上会把数据语义和实时连接都搞乱

### 纠偏

- PWA 只缓存网页壳和静态资源
- 项目数据继续走原来的浏览器本地存储
- `AnkiConnect` 这类需要实时直连本机的请求，不能交给 Service Worker 接管

## 2026-04-06：工具型 PWA 不能只做运行期零散缓存

### 现象

- 如果只是“用户访问到哪个脚本，就把哪个脚本顺手记一下”
- 那么首次打开某个功能分包时，仍然会重新去服务器拉一次
- 对工具型应用来说，这种体感还是偏慢

### 纠偏

- 构建完成后就把整套前端构建产物放进预缓存清单
- 包括主包、分包、样式和静态资源
- 后续打开网页时，优先直接走本地缓存的前端资源
- 有新版本时，再通过更新提示引导用户切换

## 2026-04-06：不能拿预览级结果去冒充最终 PNG / JPG 导出

### 现象

- 前端预览为了速度，天然会带缓存和轻量化策略
- 如果顺手拿已经生成过的 `WebP / JPG` 预览结果再转成 `PNG`
- 表面上格式变了，实际并没有回到原图链路

### 纠偏

- 导出链路必须单独判断图片是否仍是原图
- 只有明确仍是原图链路时，才开放 `PNG / JPG`
- `PNG` 导出必须从原图链路重新生成，不能复用已经压缩过的预览产物

## 2026-04-06：导出格式不能只看设置开关，要看每张图的真实来源

### 现象

- 如果只看“导入压缩设置当前开着还是关着”
- 就没法知道这张图到底是原图、导入时压过，还是中途项目压缩过
- 导出页就会出现“该锁的时候没锁，不该锁的时候也锁住”的问题

### 纠偏

- 每张图都要记录自己的来源状态
- 至少区分：
  - 原图
  - 导入压缩过
  - 项目压缩过
  - 旧版记录无法确认
- 导出格式开放规则必须基于这层真实来源，而不是只基于当前设置页

## 2026-04-06：关闭启动自动恢复时，别让自动保存顺手把旧项目抹掉

### 现象

- 产品改成“启动只提示可恢复项目，不默认自动恢复”
- 但如果自动保存仍按空白项目照常写回
- 用户还没来得及点恢复，浏览器里的旧项目就会被新的空白状态覆盖

### 纠偏

- 启动阶段先做“只探测不恢复”的摘要检查
- 只要还有待恢复的旧项目，就先暂停用空白状态覆盖本地记录
- 等用户真正恢复、清空，或开始新的导入流程后，再重新接回自动保存

## 2026-04-06：下方预览如果每次都整块重建，编辑会卡且页面会抽动

### 现象

- 每次改遮罩或裁切后，下方问题面和答案面预览都会立刻整块重渲染
- 旧预览先消失、骨架屏再出现，浏览器窗口高度也会跟着轻微抽动
- 用户会把这种延迟误判成“保存很慢”，但真正拖慢手感的往往是预览重绘

### 纠偏

- 预览更新改成低优先级，不要和主编辑动作抢同一拍
- 新预览生成时先保留旧图，只做轻量“正在更新”提示，不要先整块清空
- 预览区域高度尽量固定，避免图片重新加载时把整页高度一起带着抖动

## 2026-04-06：浏览器本地保存不能在每次编辑时整批重写所有图片

### 现象

- 明明只改了一张图的遮罩，保存日志里却总能看到整个项目的图片数量
- 项目图片一多，拖动遮罩会明显发涩
- 用户容易误以为只是浏览器限制，实际上是保存结构本身太重

### 纠偏

- 本地保存要把“图片资源”和“编辑元数据”拆开
- 图片没变时，不要每次编辑都重复写入整批 blob
- 单张图片编辑后，优先只更新对应草稿和项目摘要，再保留旧图片资源记录

## 2026-04-07：移动端聚焦编辑要保留“关闭后局部复位”的兜底

### 现象

- 移动端浏览器下，聚焦编辑偶发会残留本次操作的临时状态
- 一旦遇到这类状态串扰，用户原先往往只能刷新整页才能回到干净状态
- 这类问题未必每次都能稳定复现，也不一定能在短时间内对所有浏览器彻底根治

### 纠偏

- 移动端聚焦编辑关闭后，下次重新打开时要重建内部编辑实例
- 这层重建只清掉临时交互状态，不能把已经提交过的遮罩和裁切也一起清空
- 在真正根治浏览器手势差异前，先保留这条局部复位出口，避免用户动不动就得刷新整页

## 2026-04-08：手机 Chrome 的导出拒绝，不能只把原始报错直接弹给用户

### 现象

- 同样的网页内导出，在部分安卓浏览器里能走通，但在手机 Chrome 上会被更严格地拦下
- 顶部提示如果只原样显示浏览器抛出的 `permission denied`
- 用户只能知道“失败了”，却看不出是系统分享没打开、浏览器拦了下载，还是下载后唤起应用失败

### 纠偏

- 导出链路里的“系统分享”“浏览器下载”“下载后额外尝试打开”要分开看待
- 顶部提示优先翻译成用户能理解的原因，不要直接裸露浏览器原始英文报错
- 原始错误继续保留在控制台，方便后续判断是 Chrome 手势限制、下载限制，还是系统层拒绝

## 2026-04-08：HEIF / HEIC 进入工作台前要先统一转成可编辑格式

### 现象

- 部分手机导出的照片是 HEIF / HEIC
- 浏览器不一定能直接把这类文件走进现有的裁切、遮罩、预览和导出链路
- 如果只在某一处临时兼容，后面其他环节仍然容易继续报错

### 纠偏

- HEIF / HEIC 要在导入最前面先统一转成浏览器更稳的 JPG
- 导入完成后给用户一个轻提示，明确说明这批图已经先转格式再入场
- 转换后的图片不再算“原图链路”，后续导出格式规则要比照非原图处理，避免再把它误当成可回到原生 PNG / JPG 的来源

## 2026-04-01：聚焦模式一旦变成独立布局模式，就容易漂移

### 现象

- 出现额外空白
- 遮罩漂移
- 普通模式与聚焦模式表现不一致

### 纠偏

优先采用最上层对话层或悬浮层思路，并始终让覆盖层锚定真实图片盒子，而不是更大的布局容器。

## 2026-04-01：手动导出按 card group，而不是按单个矩形

### 现象

开发者很容易误以为“一个遮罩就一定对应一张卡”。

### 纠偏

在手动模式下，导出逻辑必须按 `card_group_id` 和 `card_order` 思考，而不是只按单个矩形遮罩思考。

## 2026-04-01：worktree 创建失败，因为当前项目还不是独立仓库

### 现象

- `fatal: invalid reference: HEAD`
- 当前仓库没有任何提交

### 纠偏

先把这个项目整理成独立 Git 仓库，并做出一个真实的基线提交，再谈 `worktree`。

## 2026-04-01：环境误拦不一定等于代码有问题

### 现象

- 读文件偶发失败
- `apply_patch` 偶发失败
- helper / 防火墙 / 沙箱相关错误看起来像随机出现

### 纠偏

先把它当作环境层噪声，再判断是否真的是业务 bug。优先采取：

1. 小补丁
2. 小范围读取
3. 分步验证
4. 必要时提权读取

## 2026-04-01：状态面板不要跟着自动保存高频弹出

### 现象

- 图片编辑时状态面板反复跳出
- 用户明明只是在画遮挡，却被“保存中”打断

### 纠偏

- 自动保存节流保持在 1 到 2 秒量级
- 状态胶囊的自动展开优先响应真正需要关注的任务，比如导出、图片导入或错误
- 不要让“保存中”这种常规后台动作抢占主交互

## 2026-04-01：导出视窗里的只读预览不能另起一套尺寸逻辑

### 现象

- 很容易为了“导出前确认”另做一套预览组件
- 一旦和主编辑器脱节，就会出现遮罩位置、裁切范围、预览结果不一致

### 纠偏

- 优先复用共享编辑器
- 通过只读模式关掉编辑动作，而不是重写一套新坐标系
- 修改共享编辑器时，继续同时回看普通编辑、聚焦编辑、只读预览三种状态

## 2026-04-01：导出视窗里的聚焦编辑不能退化成“原位放大图”

### 现象

- 如果只是把左侧预览原地切成更大的编辑视图，用户会感知成“只是放大了图片”
- 这种做法没有真正隔离注意力，反而会破坏导出页本来已经稳定的左右结构
- 若不同页面各自维护一套聚焦编辑实现，也很容易再次出现尺寸、快捷键和层级不一致

### 纠偏

- 导出页里的“聚焦编辑”应该打开真正的独立悬浮层，而不是原位放大
- 优先复用同一套聚焦编辑悬浮组件，避免手动页和导出页再次长出两种不同实现
- `Q` 只能命中当前活跃区域，其他被遮住的工作区要暂停响应
- 聚焦层必须足够宽，不能再退回成狭窄的小窗

## 2026-04-01：导出视窗要保持“三张平级卡片”，不要重新长成套娃布局

### 现象

- 虽然逻辑上已经拆成左预览、右确认、下进度，但界面上仍然容易画成“右侧大卡片里再塞一堆小卡片”
- 左侧预览区也容易重复堆叠说明、快捷键提示和多余的小预览窗，导致主次混乱

### 纠偏

- 导出页要稳定保持 3 张平级 card：
  - 左侧预览 card
  - 右侧确认 / 导出 card
  - 下方居中的小进度 card
- 底部被分离出去的，只是那条总进度条；步骤轨道仍然留在右侧主 card 顶部
- 牌组区优先做单层信息，不要把“牌组 card”再塞进“导出 card”里形成套娃感
- 自动补全只在输入框聚焦时出现，失焦就收起
- 左侧预览区不再额外显示“问题面 / 答案面”“卡片 1 / 卡片 2”这类重复确认视图

## 2026-04-02：滚轮微调遮罩不能每一步都立刻向上提交

### 现象

- 鼠标滚轮缩放遮罩时会出现明显卡顿
- 编辑中状态栏和自动保存容易被高频触发
- 虽然用户只是在微调尺寸，但主工作区会跟着频繁重渲染

### 纠偏

- 滚轮缩放时优先保证本地画面即时更新
- 向上同步和保存改成 1 到 2 秒量级的延迟提交
- 撤回快照仍然保留细粒度，但不要把每一次滚轮刻度都立刻推给外层状态
- 若用户随后执行撤回、重做、重置等明确动作，先取消旧的延迟提交，避免旧结果反写回来

## 2026-04-02：导出流程从逐张确认切到最终导出时，先别同步改全局项目

### 现象

- 第二步进入第三步明显比从第三步退回第二步更卡
- 录制里能看到不仅导出窗口在动，背后的主工作区也跟着刷新
- 这种卡顿往往出现在“确认最后一张卡片”的瞬间

### 纠偏

- 最后一张确认时，优先把牌组和标签先记在导出流程自己的本地缓存里
- 先完成左侧重预览卸载和窗口收窄，再延后写回全局项目状态
- 真正开始导出时，再统一把缓存过的编辑结果合并进目标数据
- 如果一边切阶段一边立刻改全局项目，很容易把整个页面都卷进一次不必要的重渲染

## 2026-04-02：移动端不要直接复用桌面编辑区

### 现象

- 手机或平板竖屏下，左右栏布局会严重挤压
- 常规页里直接拖遮罩，容易误触、误滚动
- 移动端拖手柄偶发卡顿，用户会误以为编辑器不稳定

### 纠偏

- 移动端竖屏改成上下布局，不再硬保留左右栏
- 常规工作区默认只读，真正编辑限制在聚焦页
- 聚焦页继续复用同一套编辑器，但关闭滚轮缩放并放大触控热区
- 如果未来再改移动端交互，先确保“只在聚焦页编辑”这条边界不要被无意打破

## 2026-04-02：移动端不应显示桌面 Anki 直连逻辑

### 现象

- 手机上展示“获取本机牌组”“检查 Anki”这类按钮，只会制造困惑
- 即使 UI 隐藏了，如果逻辑层还继续自动检测，也是在做无用功

### 纠偏

- 移动端既要隐藏相关 UI，也要跳过自动检测逻辑
- 移动端导出主路径改为 APKG
- 牌组输入改为本地牌组池，不再依赖远端或桌面本机状态

## 2026-04-02：手机通过热点访问本地页面时，别只盯着前端代码

### 现象

- 电脑上本地网页能打开，但手机通过热点或局域网访问不到
- 以为是“跨域”问题，实际上常常是开发服务根本没对外监听

### 纠偏

- 本地开发服务与预览服务都要显式监听 `0.0.0.0`
- 开发态若确实需要跨来源访问，再同时放开开发服务的 CORS
- 如果代码层已经放开，手机仍然打不开，再优先回看系统防火墙或私有网络放行

## 2026-04-02：移动端长内容卡片不要只靠省略号硬压

### 现象

- 竖屏下图片列表、导出确认这类区域很容易出现“看不全，但也滚不到”
- 只把文字改小或继续加 `truncate`，用户仍然拿不到完整信息

### 纠偏

- 长内容优先给真实滚动容器，而不是继续堆省略号
- 需要保留摘要感时，至少把移动端的关键字段改成可换行
- 若卡片处在固定高度的弹窗或工作区里，同时补上 `min-h-0`，否则滚动区域经常不会真正生效

## 2026-04-02：移动端导出弹窗不要沿用桌面的大预览和底部内嵌按钮

### 现象

- 手机上的导出弹窗如果继续展示桌面那种大图预览，会迅速挤压下方真正要操作的区域
- “上一张 / 下一张 / 确认 / 导出”如果仍塞在内容流里，用户一滚动就找不到核心动作

### 纠偏

- 移动端导出页优先改成窄条安全预览，用 `object-contain` 避免图片被裁断
- 关键操作改为底部常驻悬浮条，同时给内容区域额外预留底部空间，避免被悬浮条遮住
- 图片统计和状态信息改成紧凑胶囊，不要继续用大号统计卡片挤占竖屏空间

## 2026-04-02：APKG 导出不能只改预览，不改真正打包图片

### 现象

- 导出面板里虽然有“压缩质量”，但最终生成的 APKG 仍然特别大
- 用户会误以为质量调节失效，实际上是打包那条链路还在用默认的大图格式

### 纠偏

- 直连导入与 APKG 打包都要显式接入同一份质量参数
- 不要默认继续输出无压缩的大 PNG，卡包导出至少要走受质量控制的压缩格式
- 每次改导出面板后，都要回看“预览图”“直连导出”“APKG 打包”三条线是否一致

## 2026-04-03：右上角提示层级过低时，AnkiConnect 引导会被挡住

### 现象

- 首次 Anki 直连失败时，右上角会弹出一个轻提醒
- 但它可能和别的提示重叠，结果最需要被看到的那条反而被压在下面

### 纠偏

- 右上角消息提示的层级要显式抬高，不要默认相信组件库的初始层级
- 如果这条提示承担“把用户引到帮助入口”的职责，就必须优先保证它能完整露出来
- 同时把关闭按钮收成轻量小叉，避免视觉噪声盖过真正的提示内容

## 2026-04-03：桌面端的说明提示不要误做成“点击型”

### 现象

- 文本里的轻量强调块，桌面端本来只是想悬停显示说明
- 但如果把它做成按钮，就容易和真正可点击的帮助入口串扰
- 结果会出现“点 AnkiConnect，却把 APKG 说明顺带触发出来”这种错觉

### 纠偏

- 桌面端的纯说明型提示优先走悬停，不要额外做成点击触发
- 真正需要点击的只保留给明确动作，例如打开帮助
- 同一个外观组件如果同时承担“说明”和“跳转”两类角色，要在交互层明确分开

## 2026-04-03：导入预处理、Anki 导出、APKG 打包不能各走各的图片规则

### 现象

- 用户看到的编辑图和最终导出的图，可能不是同一份尺寸
- 超高分辨率图片下，挖空描边会显得特别细
- 很容易只改纯图像组，忘了 Anki / APKG 仍然在走固定 WebP

### 纠偏

- 图片进入工作台时，先统一做压缩和最大分辨率校正
- 预览、编辑、Anki 导出、APKG 打包都继续复用这份处理后的图片
- 当前只有“纯图像组”允许切换 `WebP / JPG / PNG`
- `AnkiConnect` 与 `APKG` 两条线继续固定为 `WebP`，不要误以为它们也会跟着设置切换格式

## 2026-04-03：shadcn 的 DialogContent 默认会把桌面端宽度卡成窄窗

### 现象

- 明明已经给弹窗写了更大的 `w` 或 `max-w`
- 但桌面端看起来仍然像一条很窄的弹窗
- 特别容易出现在设置页这类需要横向排布的场景里

### 根因

- 当前项目的 `DialogContent` 底层默认带了 `sm:max-w-sm`
- 如果业务层只写 `max-w-[76rem]` 这类类名，而没有显式覆写 `sm:max-w-*`
- 桌面端最终还是会被这个默认窄宽度卡住

### 纠偏

- 改宽弹窗时，优先参考项目里已经正常工作的其他弹窗写法
- 在业务层显式覆写 `sm:max-w-*`，不要只写基础 `max-w-*`
- 如果弹窗需要做桌面端左右排布，先确认真正生效的不是组件底层那个默认 `sm:max-w-sm`

## 2026-04-03：为了桌面端垂直居中，不要直接给移动端列容器加 `self-center`

### 现象

- 设置页这类竖排弹窗里，某一张卡片会突然被压成很窄的一条
- 内容本身没坏，但整张卡的可用宽度被莫名收窄

### 根因

- 在 `flex-col` 的移动端容器里，如果给子列直接加 `self-center`
- 子列就不再默认撑满父容器，而会按自身内容宽度收缩
- 桌面端为了两栏垂直居中而加上的这类对齐，很容易误伤移动端

### 纠偏

- 移动端优先保持 `w-full` 和 `self-stretch`
- 只在桌面端断点上再补 `md:self-center`
- 需要“桌面端平衡、移动端满宽”时，优先把对齐限制在对应断点里，不要一套类名同时作用两端

## 2026-04-03：移动端编辑拖动不要直接跟着原始 `pointermove` 频率整块刷新

### 现象

- 手机上拖动遮罩或裁切手柄时，会先顿一下再跟手
- 图片越大、项目里图片越多，这种顿挫越明显

### 根因

- 原始 `pointermove` 触发频率很高
- 如果每来一个事件就立刻整块更新编辑区，移动端主线程很容易被挤满
- 再叠加大图重绘和缩略图解码，卡顿会进一步放大

### 纠偏

- 拖动中的视觉更新优先按帧合并，不要直接吃满原始事件频率
- 高频拖动时，先保证手感跟手，再在结束时统一提交结果
- 列表缩略图、预览图这类非主编辑图，优先走惰性加载和异步解码，减少同屏重资源竞争

## 2026-04-03：导入压缩一旦开启，导出格式选择必须跟着收窄

### 现象

- 如果图片在导入时已经先被有损压缩过，再让用户随意切 `PNG / JPG / WebP`
- 很容易让人误以为“换成 PNG 就能把细节补回来”

### 纠偏

- 导出格式联动规则保持一致：
  - 只要导入时已经压缩，三条导出路径都统一锁住 `PNG / JPG`
  - 只要中途执行过项目压缩，也统一锁住 `PNG / JPG`
  - 只有明确仍是原图链路时，才开放 `WebP / JPG / PNG`
- 不要再让 `AnkiConnect`、`APKG`、纯图像组三条线各自长出不同规则

## 2026-04-03：站内危险操作优先用项目内确认窗，浏览器原生确认只留给离开页面

### 现象

- 清空项目、重置当前图片这类动作，如果直接用浏览器原生弹窗，会和整体 UI 风格脱节
- 但刷新页面、侧滑返回、误触关闭标签页这类离开页面动作，又必须依赖浏览器原生保护

### 纠偏

- 站内的危险操作优先复用统一的确认弹窗组件，不要继续散落使用原生 `confirm`
- 浏览器原生离开页面提示只保留给“还有未导出编辑结果时阻止误离开”这一类场景
- 后续再增加确认类交互时，优先回看现有弹窗组件，不要重新搓一套样式和逻辑

## 2026-04-03：图片列表缩略图的加载态不能跟着任何草稿变动一起清空

### 现象

- 画完遮罩或自动保存之后，左侧缩略图会突然白掉
- 不是图片资源真的丢了，而是白底和骨架一直盖在上面

### 根因

- 如果把“缩略图是否已加载完成”的状态直接绑定到整个 `items` 数组
- 那么只要草稿里任意字段变化，例如遮罩数量、牌组、保存时间
- 这批状态就会被整包清空
- 但缩略图地址又没有真的变化，于是图片不会重新触发一次加载完成事件

### 纠偏

- 缩略图加载态只应在图片本身变了时重置，例如图片 id 或图片地址变化
- 不要因为遮罩、标签、保存状态变化就把整批缩略图重新打回未加载
- 这类缩略图组件后续若再优化，优先回看“图片资源变化”和“草稿数据变化”是不是被混成一层了

## 2026-04-03：图片列表项不能再用按钮套按钮

### 现象

- 左侧图片列表点击时，控制台出现 `<button> cannot be a descendant of <button>`
- React 会继续报警告，甚至可能连带引出水合和焦点问题

### 根因

- 整个列表项如果本身已经是一个 `button`
- 再把删除按钮、更多按钮之类的小动作按钮塞进去
- 最终就会生成按钮嵌套按钮的非法结构

### 纠偏

- 这类“整行可点、行内还有小按钮”的列表项，外层优先改成带 `role="button"` 的普通容器
- 真正的小动作继续保留为标准按钮
- 后续再改图片列表、牌组列表、弹窗列表时，先检查 DOM 结构，不要只看视觉上能不能点

## 2026-04-03：分栏宽度记忆不能在每次布局回调里无脑写回状态

### 现象

- 页面刚加载或拖动分栏后，React 报 `Maximum update depth exceeded`
- 表面上像是“记住宽度”失效，实际上是状态和布局在互相来回触发

### 根因

- 分栏组件每次布局变化都会继续发回当前尺寸
- 如果外层不判断“这次尺寸和上一次是否真的变了”，就会每次都写状态
- 状态一写回，又会触发布局回调，最后形成更新死循环

### 纠偏

- 布局记忆要先比较新旧值，真的变了再写回
- 本地缓存只记最终稳定尺寸，不要把每次相同回调都当成新变化
- 以后再给分栏、拖拽面板做“记住上次宽度”时，先确认有没有这一层去重保护

## 2026-04-03：分栏组件吃的是比例，不是像素

### 现象

- 代码里明明想表达“左栏最少保留 340 像素”
- 但如果把 `340` 直接塞给分栏组件的最小值，页面会表现得非常怪，甚至可能把布局拖进异常状态

### 根因

- 当前这套分栏底层接收的是百分比尺寸
- 直接把像素值喂进去，等于把“340 像素”误写成了“340%”

### 纠偏

- 常量 `340` 可以继续保留为“真实像素下限”
- 但要先根据当前容器宽度换算成比例，再交给分栏组件
- 后续再调默认宽度、最小宽度时，先分清“产品想表达的是像素”还是“组件实际吃的是比例”

## 2026-04-03：滚动定位不要在列表 map 里反复创建回调 ref

### 现象

- 导出弹窗里的缩略图队列，表面只是想在切图时自动滚到当前项
- 但如果给每个列表项都在渲染时临时创建一份新的回调 ref
- 就容易和 Radix / ScrollArea 的内部 ref 组合在一起，触发反复 setState

### 纠偏

- 这类“切换当前项后滚到可见区域”的需求，优先用容器查询或稳定 ref
- 不要在 `map` 里给每个子项都现生成一份新的回调 ref
- 尤其在弹窗、滚动区、动画区叠加时，先怀疑这类不稳定引用是不是把底层组件拖进了循环

## 2026-04-08：HEIF / HEIC 不要和常规图片走同一拍导入完成语义

### 现象

- 用户一次选中常规图片和 `HEIF / HEIC` 时
- 如果等全部一起转完才入列，导入等待会明显变长
- 如果把 `HEIF / HEIC` 直接当作已就绪图片放进编辑区，又会出现无法编辑、后缀不对或误以为已经可导出的错觉

### 根因

- `HEIF / HEIC` 进入网页编辑前，本质上先要做一次格式转换
- 这一步常常要 1 到 5 秒，不能假装成和 `jpg / png / webp` 一样即时完成
- 同时，转换结果的文件名后缀、导出开放规则和编辑就绪状态，也必须一起切到转换后的真实语义

### 纠偏

- 若同批次里混有 `HEIF / HEIC`，先优先处理常规图片
- `HEIF / HEIC` 要先以“转换中”占位进入图片列表，完成后再开放编辑
- 转换结果的显示后缀要立即改成真实落地后缀，例如 `.jpg`
- 转换与后续导入压缩，都要跟随当前设置，不要再额外走一套隐藏默认值

## 2026-04-09：首屏动画不要靠“先挂载再隐藏”伪装延迟

### 现象

- 首屏明明要求先只显示主屏内容
- 但如果把编辑区、图片选择区、底部导出条提前挂上，再靠透明度或位移动画把它们“藏住”
- 用户仍然会感觉页面一开始就已经摆满，动画像是盖在现成页面上的假动作

### 根因

- 共享布局动画只负责把标题、图标、主按钮从主屏顺到工作台
- 工作台主体如果提前挂载，会和首屏动画同时争夺注意力
- 顶部通知、状态胶囊、工作台内部内容若不一起做时序隔离，也会把主屏压碎

### 纠偏

- 首屏阶段优先只挂主屏，不要提前挂工作台主体
- 工作台主体要在明确延迟后再真正挂载，而不是仅靠 `opacity: 0` 伪装
- 页头里承担共享布局的元素单独保留，其他说明、按钮组、模式切换等内容再延后淡入
- 顶部通知和状态提示在首屏与工作台引导动画阶段都应暂时收起，避免覆盖主屏
