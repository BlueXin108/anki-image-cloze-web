# 踩坑与纠偏记录

## 2026-04-01：聚焦模式一旦变成独立布局模式，就容易漂移

### 现象

- 出现额外空白
- 遮罩漂移
- 普通模式与聚焦模式表现不一致

### 纠偏

优先采用最上层对话层或悬浮层思路，并始终让覆盖层锚定真实图片盒子，而不是更大的布局容器。

## 2026-04-01：手动导出按 card group，而不是按单个矩形

### 现象

开发者很容易误以为“一个遮罩就一定对应一张卡”。

### 纠偏

在手动模式下，导出逻辑必须按 `card_group_id` 和 `card_order` 思考，而不是只按单个矩形遮罩思考。

## 2026-04-01：worktree 创建失败，因为当前项目还不是独立仓库

### 现象

- `fatal: invalid reference: HEAD`
- 当前仓库没有任何提交

### 纠偏

先把这个项目整理成独立 Git 仓库，并做出一个真实的基线提交，再谈 `worktree`。

## 2026-04-01：环境误拦不一定等于代码有问题

### 现象

- 读文件偶发失败
- `apply_patch` 偶发失败
- helper / 防火墙 / 沙箱相关错误看起来像随机出现

### 纠偏

先把它当作环境层噪声，再判断是否真的是业务 bug。优先采取：

1. 小补丁
2. 小范围读取
3. 分步验证
4. 必要时提权读取

## 2026-04-01：状态面板不要跟着自动保存高频弹出

### 现象

- 图片编辑时状态面板反复跳出
- 用户明明只是在画遮挡，却被“保存中”打断

### 纠偏

- 自动保存节流保持在 1 到 2 秒量级
- 状态胶囊的自动展开优先响应真正需要关注的任务，比如导出、图片导入或错误
- 不要让“保存中”这种常规后台动作抢占主交互

## 2026-04-01：导出视窗里的只读预览不能另起一套尺寸逻辑

### 现象

- 很容易为了“导出前确认”另做一套预览组件
- 一旦和主编辑器脱节，就会出现遮罩位置、裁切范围、预览结果不一致

### 纠偏

- 优先复用共享编辑器
- 通过只读模式关掉编辑动作，而不是重写一套新坐标系
- 修改共享编辑器时，继续同时回看普通编辑、聚焦编辑、只读预览三种状态

## 2026-04-01：导出视窗里的聚焦编辑不能退化成“原位放大图”

### 现象

- 如果只是把左侧预览原地切成更大的编辑视图，用户会感知成“只是放大了图片”
- 这种做法没有真正隔离注意力，反而会破坏导出页本来已经稳定的左右结构
- 若不同页面各自维护一套聚焦编辑实现，也很容易再次出现尺寸、快捷键和层级不一致

### 纠偏

- 导出页里的“聚焦编辑”应该打开真正的独立悬浮层，而不是原位放大
- 优先复用同一套聚焦编辑悬浮组件，避免手动页和导出页再次长出两种不同实现
- `Q` 只能命中当前活跃区域，其他被遮住的工作区要暂停响应
- 聚焦层必须足够宽，不能再退回成狭窄的小窗

## 2026-04-01：导出视窗要保持“三张平级卡片”，不要重新长成套娃布局

### 现象

- 虽然逻辑上已经拆成左预览、右确认、下进度，但界面上仍然容易画成“右侧大卡片里再塞一堆小卡片”
- 左侧预览区也容易重复堆叠说明、快捷键提示和多余的小预览窗，导致主次混乱

### 纠偏

- 导出页要稳定保持 3 张平级 card：
  - 左侧预览 card
  - 右侧确认 / 导出 card
  - 下方居中的小进度 card
- 底部被分离出去的，只是那条总进度条；步骤轨道仍然留在右侧主 card 顶部
- 牌组区优先做单层信息，不要把“牌组 card”再塞进“导出 card”里形成套娃感
- 自动补全只在输入框聚焦时出现，失焦就收起
- 左侧预览区不再额外显示“问题面 / 答案面”“卡片 1 / 卡片 2”这类重复确认视图

## 2026-04-02：滚轮微调遮罩不能每一步都立刻向上提交

### 现象

- 鼠标滚轮缩放遮罩时会出现明显卡顿
- 编辑中状态栏和自动保存容易被高频触发
- 虽然用户只是在微调尺寸，但主工作区会跟着频繁重渲染

### 纠偏

- 滚轮缩放时优先保证本地画面即时更新
- 向上同步和保存改成 1 到 2 秒量级的延迟提交
- 撤回快照仍然保留细粒度，但不要把每一次滚轮刻度都立刻推给外层状态
- 若用户随后执行撤回、重做、重置等明确动作，先取消旧的延迟提交，避免旧结果反写回来

## 2026-04-02：导出流程从逐张确认切到最终导出时，先别同步改全局项目

### 现象

- 第二步进入第三步明显比从第三步退回第二步更卡
- 录制里能看到不仅导出窗口在动，背后的主工作区也跟着刷新
- 这种卡顿往往出现在“确认最后一张卡片”的瞬间

### 纠偏

- 最后一张确认时，优先把牌组和标签先记在导出流程自己的本地缓存里
- 先完成左侧重预览卸载和窗口收窄，再延后写回全局项目状态
- 真正开始导出时，再统一把缓存过的编辑结果合并进目标数据
- 如果一边切阶段一边立刻改全局项目，很容易把整个页面都卷进一次不必要的重渲染
